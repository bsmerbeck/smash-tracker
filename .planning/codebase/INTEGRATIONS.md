# External Integrations

**Analysis Date:** 2026-07-09

## APIs & External Services

**Tournament Data (Start.gg):**

- start.gg GraphQL API - Tournament event/entrant/set data sync
  - SDK/Client: fetch (Node.js built-in, injected for testing)
  - Auth: Bearer token (`STARTGG_API_TOKEN` for public data reads)
  - OAuth: start.gg OAuth 2.0 for user account linking (`STARTGG_CLIENT_ID`, `STARTGG_CLIENT_SECRET`)
  - Implementation: `apps/api/src/startgg/client.ts`, `apps/api/src/routes/startgg.ts`
  - Routes: `/api/integrations/startgg/status`, `/api/integrations/startgg/authorize`, `/api/integrations/startgg/sync`, `/auth/startgg/login`, `/integrations/startgg/callback`

**Tournament Data (Parry.gg):**

- parry.gg gRPC-Web API - Match history, player profiles, character data
  - SDK/Client: `@parry-gg/client` (autogenerated from .proto)
  - Transport: gRPC-Web via `grpc-web` (XMLHttpRequest polyfill via `xhr2` in Node)
  - Auth: X-API-KEY header (`PARRYGG_API_KEY`)
  - Endpoint: `https://grpcweb.parry.gg`
  - Implementation: `apps/api/src/parrygg/client.ts`
  - Routes: `/api/integrations/parrygg/status`, `/api/integrations/parrygg/link`, `/api/integrations/parrygg/sync`, `/auth/parrygg/login-start`, `/auth/parrygg/login-complete`, `/auth/parrygg/verification-start`, `/auth/parrygg/verification-complete`

**AI Report Generation:**

- Anthropic Claude API - AI-powered scouting report generation
  - SDK/Client: `@anthropic-ai/sdk` (version 0.110.0)
  - Auth: Bearer token (`ANTHROPIC_API_KEY`)
  - Implementation: `apps/api/src/reports/generate.ts`, `apps/api/src/routes/reports.ts`
  - Routes: `POST /api/reports` (generate), `GET /api/reports/:id` (retrieve), `GET /api/reports/config` (availability check)
  - Gating: Allowlist (`REPORTS_ALLOWED_UIDS`) or pay-per-use credits (Stripe optional)

## Data Storage

**Databases:**

- Firebase Realtime Database (RTDB)
  - Connection: `FIREBASE_DATABASE_URL` (required, passed to firebase-admin)
  - Client: `firebase-admin/database` (server) + direct RTDB emulator support via `FIREBASE_DATABASE_EMULATOR_HOST`
  - Schema: User-keyed hierarchy (`matches/:uid`, `opponents/:uid`, `groups/`, `groupMembers/`, `userGroups/`, `startggLinks/:uid`, `parryggLinks/:uid`, `groupInviteCodes/`, `scoutReports/:uid`, `scoutCache/:uid`, `gspLive/`)
  - Auth: Firebase ID tokens (JWT from Firebase Auth)

**File Storage:**

- Firebase Storage (implicit via Hosting)
  - Storage rules: `storage.rules`
  - Used for: OG image cards generated at build time, hosted via Hosting static assets

**Caching:**

- Redis: Not detected — caching is in-process (Vitest cache) or browser-based (React Query)
- Browser cache: `Cache-Control` headers via Firebase Hosting (`apps/web/dist/`, 1 year for `/assets/**`, no-cache for root)

## Authentication & Identity

**Auth Provider:**

- Firebase Authentication
  - Sign-in methods: Google OAuth (implicit via Google provider)
  - Custom token flow: Start.gg OAuth callback generates Firebase custom tokens for "Login with start.gg"
  - Implementation:
    - Web SDK: `apps/web/src/lib/firebase.ts`, `apps/web/src/context/AuthContext.tsx`
    - Server: `apps/api/src/plugins/auth.ts` (ID token verification via firebase-admin)

**OAuth/Third-party:**

- Start.gg OAuth 2.0: `STARTGG_CLIENT_ID`, `STARTGG_CLIENT_SECRET`, `STARTGG_REDIRECT_URI`
  - State signing/verification: HMAC-SHA256 with `STARTGG_STATE_SECRET`
  - Flows: Link account (authenticated user) + Login (new user or existing)
  - Callback: `/integrations/startgg/callback` (verifies state, exchanges code, creates/finds Firebase user)

- Parry.gg: No OAuth — bio-text verification code + X-API-KEY
  - Link flow: User must include a verification code in their parry.gg bio; server reads bio via API to confirm ownership
  - Implementation: `apps/api/src/parrygg/verificationCode.ts`

## CI/CD & Deployment

**Hosting:**

- Firebase Hosting + Cloud Run (serverless containers)
- Architecture: Hosting static assets + rewrites `/api/**` to Cloud Run service (`smash-tracker-api`, region `us-central1`)
- Deployment target: GCP (Firebase Realtime Database, Firebase Auth, Firebase Hosting)

**CI Pipeline:**

- Not detected in codebase (likely GitHub Actions configured externally)
- Dockerfile multi-stage build: `deps` → `build` → `runtime`
  - Optimized with `pnpm deploy --prod` for production image (no devDeps, no apps/web)
  - Environment: `NODE_ENV=production`
  - Node image: `node:24-slim` (smaller runtime)

## Environment Configuration

**Required env vars:**

- `FIREBASE_DATABASE_URL` - Realtime Database instance URL (always required)
- `CORS_ORIGIN` - Comma-separated allowed origins for API CORS (defaults to `http://localhost:5173`)
- `WEB_BASE_URL` - SPA origin for OAuth redirects (defaults to `http://localhost:5173`)

**Optional feature flags:**

- Start.gg integration: `STARTGG_CLIENT_ID`, `STARTGG_CLIENT_SECRET`, `STARTGG_REDIRECT_URI`, `STARTGG_API_TOKEN`, `STARTGG_STATE_SECRET`
- AI reports: `ANTHROPIC_API_KEY`, `REPORTS_ALLOWED_UIDS`
- Stripe billing: `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`
- Parry.gg integration: `PARRYGG_API_KEY`

**Optional firebase-admin config:**

- `GOOGLE_APPLICATION_CREDENTIALS` - Path to service account JSON (file-based auth; omitted when running on Cloud Run with workload identity)
- `FIREBASE_DATABASE_EMULATOR_HOST` - Emulator address for local development (e.g., `127.0.0.1:9000`)

**Optional web SDK config (Vite env vars):**

- `VITE_FIREBASE_API_KEY`
- `VITE_FIREBASE_AUTH_DOMAIN`
- `VITE_FIREBASE_PROJECT_ID`
- `VITE_FIREBASE_APP_ID`
- `VITE_FIREBASE_MEASUREMENT_ID` (optional — analytics skipped if unset)

**Secrets location:**

- GCP Secret Manager or Cloud Run environment variables (production)
- `.env` or shell environment (local development)

## Webhooks & Callbacks

**Incoming:**

- `POST /api/billing/webhook` - Stripe webhook (signature verified via `stripe-signature` header + `STRIPE_WEBHOOK_SECRET`)
  - Events: `checkout.session.completed` (credit purchase confirmation)
  - Raw body required (handled via fastify raw-body parser)

**Outgoing:**

- Start.gg OAuth callback: Server exchanges code + redirects user back to `WEB_BASE_URL` with custom token fragment
- Stripe checkout: Stripe redirects user to `webBaseUrl` after payment
