<objective>
Ship the tag data model: an optional `tags: string[]` on the match record and on each `vodTimestamps` entry, following the exact `vodStartSeconds` schema-evolution playbook (optional field + conditional-spread write). No new endpoints, no tags tree — tags are embedded arrays that ride the existing full-overwrite PATCH.

Purpose: This is the backend enabler that ALL Phase 3 web slices depend on. Per the twice-learned production lesson (2026-07-10), preview channels hit the PROD API and zod silently strips unknown keys, so this schema+passthrough must reach master and deploy to prod BEFORE any preview-channel human check. This plan is therefore a self-contained, cherry-pickable unit: it touches ONLY `packages/shared` and `apps/api` — zero web files — so the orchestrator can PR its diff to master and deploy prod (rev 00034+) before the web work lands.

Output: Match- and note-level `tags` accepted, validated (per-tag ≤24 chars trimmed nonempty, ≤10/match, ≤5/note), and persisted; API tests proving store/omit/empty-drop behavior.

MVP-mode note: Under MVP vertical-slice rules the first task is normally a failing end-to-end test. This plan is the documented exception — a genuine deploy-ordering prerequisite (the prod-API deploy gate) that the phase constraints and the roadmap's cherry-pick-to-master requirement mandate be its own backend-only unit. The first USER-FACING vertical slice is 03-02 (match tagging), which builds directly on the schema shipped here.
</objective>

**As a** SSBU player with a growing library of recorded VODs, **I want to** tag matches and individual timestamp notes with preset and freeform tags and filter my VOD list by those tags, **so that** my footage stays organized the way I actually think about my play and I can quickly narrow to what I'm looking for.
